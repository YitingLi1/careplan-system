## 📚 今天学到的核心概念

### 1. 需求澄清（面试高频考点）

**为什么重要：** 面试官拿到需求后第一步不是写代码，而是问问题。
直接写代码是最常见的错误。

**你学会的方法：**
1. 通读一遍，不停下来
2. 找"模糊词"（Warning / Only once / Quick way）
3. 把模糊词变成具体问题

**面试时怎么说：**
> "拿到需求后我会先通读一遍，标出描述不够具体的地方，
> 然后列出需要和客户确认的问题，特别关注边界情况和错误处理方式。"

---

### 2. 前后端双层验证

**记住这句话：**
> 前端验证是为了用户体验，后端验证是为了数据安全。两个都必须做，因为前端验证可以被绕过。

| 层 | 做什么 | 为什么 |
|---|---|---|
| 前端 | 格式检查、必填检查 | 即时反馈，不用等服务器 |
| 后端 | 格式再检查 + 业务逻辑验证 | 防止用户绕过前端直接调 API |

---

### 3. 数据模型设计思路

**核心原则：** 字段放在"会随情况变化"的那个实体里。

**你今天改正的错误：**

❌ 错误版本：diagnosis 放在 Patient 里
✅ 正确版本：diagnosis 放在 Order 里

**原因：** 同一个病人不同次开药，diagnosis 可能不同。
diagnosis 是"这次开药"的信息，不是"这个病人"的信息。

**实体关系：**
```
Patient 1 --- N Order
Provider 1 --- N Order
Order   1 --- 1 CarePlan
```

---

### 4. .gitignore 是什么

**一句话：** 告诉 Git 哪些文件不要追踪、不要上传到 GitHub。

**为什么需要：**
- `.env` 里有数据库密码、API key，绝对不能上传
- `__pycache__/` 是编译缓存，没必要上传
- `.DS_Store` 是 Mac 系统文件，跟代码无关

**这个项目加了：**
```
__pycache__/   # Python 缓存
*.pyc          # 编译文件
.env           # 环境变量（密钥！）
venv/          # 虚拟环境
.DS_Store      # Mac 系统文件
.vscode/       # 编辑器配置
```

---

### 5. Git 三个核心命令

```bash
git add .                    # 把所有改动加到暂存区
git commit -m "说明"          # 提交，写说明
git push                     # 推到 GitHub
```

**什么时候 commit：** 代码能跑了就 commit。
**为什么：** 改乱了可以用 `git checkout .` 回到上一个能跑的版本。

## ⚠️ 今天不确定 / 容易忘的地方

### 问题 1：需求文档里没写到的东西怎么办？

比如今天"登录系统"需求文档里没提，怎么处理？

**答：** 两种方式：
1. 问客户确认
2. 做合理假设，写进 Design Doc，等客户确认

今天的假设：Phase 1 不做登录，所有人共用系统。

---

### 问题 2：care plan 对应病人还是订单？

**答：一个 care plan 对应一个订单（一种药物）。**

原因：同一个病人可能同时开多种药，每种药需要单独的 care plan。
这个决定直接影响数据库设计（Order 和 CarePlan 是一对一关系）。

---

### 问题 3：Design Doc 的数据模型部分

容易犯的错：把跟"这次开药"相关的字段放到 Patient 里。

**检查方法：** 问自己"同一个病人开第二次药，这个字段会变吗？"
- 会变 → 放 Order
- 不会变 → 放 Patient
